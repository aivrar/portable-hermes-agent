# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

`portable-hermes-agent` is a **Windows-portable fork** of [NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent) bundled with an embedded Python runtime, a Tkinter desktop GUI (`gui/`), and additional tools for LM Studio, TTS, Music, and ComfyUI extension servers. End users run it with no admin rights, no system Python, and no Docker.

Two pre-existing development guides cover the upstream framework in depth and are authoritative for everything not specific to the portable bundle:

- `AGENTS.md` — `AIAgent` loop, tool registry, slash command registry, skin engine, prompt-caching policy, known pitfalls.
- `CONTRIBUTING.md` — skill-vs-tool decision, full project structure, security model, cross-platform rules, PR/commit conventions.

Read those before architectural changes. The notes below cover what's different about this fork.

## Running Python in this repo

There is **no active venv**. The portable Python is at `python_embedded/python.exe` and is the interpreter every launcher and tool uses. AGENTS.md begins with `source venv/bin/activate` — that instruction is from upstream and does not apply here.

To run any Python command from a bash shell:

```bash
./run_py.sh -m pytest tests/ -q                  # full test suite
./run_py.sh -m pytest tests/test_model_tools.py  # single test file
./run_py.sh -m hermes_cli.main <subcommand>      # invoke the CLI directly
```

`run_py.sh` is a one-line wrapper that calls `python_embedded/python.exe`. It's regenerated by `install.bat` (step 6b) with LF line endings — don't let an editor convert it to CRLF or it stops working under Git Bash. `run_py.bat` is the cmd-shell equivalent but is hardcoded to `E:\hermes\...` and is not used here.

## Launchers (Windows batch entry points)

| Script | Purpose |
|--------|---------|
| `START.bat` | First-time setup + GUI launch. Calls `install.bat` if `python_embedded\python.exe` is missing, then launches `gui/app.py`. |
| `install.bat` | Full bootstrap: downloads embedded Python 3.13.12, Tcl/Tk, pip, all deps via `pip install -e .[messaging,cron,cli,mcp,honcho,pty,tts-premium,homeassistant]`, mini-swe-agent, Pillow/ddgs/lmstudio. Initializes git submodules. Syncs skills to `~/.hermes/skills/`. Re-runnable. |
| `hermes.bat` | CLI launcher. Sets `PATH`, `PIP_TARGET`, `PYTHONPATH` to the embedded Python; sets `chcp 65001` and `PYTHONIOENCODING=utf-8`; **clears stray `OPENAI_*`/`ANTHROPIC_*` env vars** so external shell auth doesn't shadow `~/.hermes/.env`; calls `python -m hermes_cli.main`. |
| `hermes_gui.bat` / `hermes_gui.vbs` | GUI launcher. Same env setup as `hermes.bat` plus `TCL_LIBRARY`/`TK_LIBRARY`, then runs `gui/app.py`. The `.vbs` variant launches without a console window. |

If you change PATH/env handling in one launcher, mirror it in the others — they share the same intent.

## Configuration layout

This fork uses repo-local config files **in addition to** the standard `~/.hermes/` user dir:

| Location | Purpose |
|----------|---------|
| `./.env` | API keys (copied from `.env.example` on install). Repo-local — not in `~/.hermes/.env`. |
| `./cli-config.yaml` | CLI settings (copied from `cli-config.yaml.example` on install). The installer post-processes this file to replace U+2014 `—` and U+2192 `→` because cp1252 console output mangles them. |
| `~/.hermes/permissions.json` | Per-user permissions: `read/write/install/execute/remove/network` levels. Created with safe defaults by `install.bat`. |
| `~/.hermes/skills/` | Skills are synced here from `./skills/` by `tools/skills_sync.py` during install. |
| `~/.hermes/SOUL.md` | Personality file. `hermes.bat` copies `assets/SOUL.md` here on first run if missing. |

Both `.env` and `cli-config.yaml` are gitignored. Don't commit them.

## Desktop GUI (`gui/`)

The Tkinter GUI is **specific to this fork** — it does not exist upstream. Entry point is `gui/app.py:main`. Architecture:

```
gui/app.py            # Tk root, chat surface, message bubbles, sidebar
 ├── agent_bridge.py  # Threaded wrapper around AIAgent — keeps Tk responsive
 ├── api_setup_wizard.py  # First-run key collection (OpenRouter, Anthropic, ...)
 ├── extensions.py    # TTS / Music / ComfyUI server lifecycle
 ├── lm_studio.py     # LM Studio model load/unload panel
 ├── permissions_panel.py + permissions.py  # ~/.hermes/permissions.json editor
 └── theme.py         # Dark theme colors, fonts, DPI scaling
```

The GUI talks to the agent via `AgentBridge` (threading + queue), never directly. When changing agent behavior, verify the GUI still works — type-check passes are not enough.

## Fork-specific tools

These tool files exist on top of upstream and are loaded through the same self-registering pattern (see `tools/registry.py` and the `_modules` list in `model_tools.py`):

- `tools/lm_studio_tools.py` — local LM Studio SDK + HTTP control (10 tools).
- `tools/tts_tool.py`, `tools/transcription_tools.py`, `tools/voice_mode.py`, `tools/neutts_synth.py` — TTS/STT/voice.
- `tools/image_generation_tool.py` — ComfyUI bridge.
- `tools/workflow_tool.py` — multi-step automation pipelines.
- `tools/tool_maker.py` — runtime tool creation (REST wrappers + Python handlers).
- `tools/serper_search_tool.py`, `tools/gpu_tool.py`, `tools/guide_tool.py`, `tools/model_switcher_tool.py`, `tools/update_hermes_tool.py`.

Schema/handler/registration co-location and the 3-file edit pattern (tool file + `model_tools.py` import + `toolsets.py` membership) are described in AGENTS.md and CONTRIBUTING.md — follow that pattern for any new tool.

## Submodules

```
mini-swe-agent/   → SWE-agent/mini-swe-agent (terminal-execution backend)
tinker-atropos/   → nousresearch/tinker-atropos (RL training)
```

`install.bat` runs `git submodule update --init --recursive`. If you clone manually, use `git clone --recurse-submodules` or follow up with `git submodule update --init`. Both are imported as editable packages.

## Tests

```bash
./run_py.sh -m pytest tests/ -q                          # ~3000 tests, runs with -n auto
./run_py.sh -m pytest tests/test_model_tools.py -q       # toolset resolution
./run_py.sh -m pytest tests/test_cli_init.py -q          # CLI config loading
./run_py.sh -m pytest tests/gateway/ -q
```

`pyproject.toml` sets `addopts = "-m 'not integration' -n auto"`, so integration tests are skipped by default and the suite runs in parallel via pytest-xdist. The `_isolate_hermes_home` autouse fixture in `tests/conftest.py` redirects `HERMES_HOME` to a temp dir — never hardcode `~/.hermes/` paths in tests.

## Critical policies inherited from upstream

These are the fork-relevant excerpts from AGENTS.md — do **not** violate them:

- **Prompt caching must not break.** Don't alter past context, swap toolsets, or rebuild system prompts mid-conversation. The only sanctioned mutation is context compression.
- **Schema descriptions must not name tools from other toolsets.** Cross-references are added dynamically in `get_tool_definitions()` in `model_tools.py` — see the `browser_navigate` / `execute_code` post-processing blocks for the pattern.
- **`_last_resolved_tool_names` is a process-global** in `model_tools.py`. `delegate_tool.py` saves/restores it around child runs; new readers must tolerate temporary staleness.
- **Don't use `\033[K`** in spinner/display code — leaks as literal `?[K` under `prompt_toolkit.patch_stdout`. Pad with spaces instead.

## Windows-specific gotchas in this bundle

- `cli-config.yaml.example` contains em-dashes and arrows. The installer rewrites them on copy so the cp1252 console doesn't fall over. If you add new files copied during install, keep them ASCII-safe or apply the same rewrite.
- The launchers proactively unset `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `ANTHROPIC_API_KEY`, `ANTHROPIC_TOKEN` so a user's global LM Studio / Codex / Anthropic-CLI env doesn't override `~/.hermes/.env`. If you add a new provider, consider adding its env var to that purge list in `hermes.bat`.
- `python_embedded/`, `python_embedded.zip`, `node_modules/`, and the `extensions/*` clones are gitignored. Don't commit them.
- Edit `run_py.sh` only with LF endings.
